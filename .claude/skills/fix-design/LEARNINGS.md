# Fix-design learnings — Responsive UX/UI Portfolio

Living reference for the `fix-design` skill. Update in place, don't append a changelog — see SKILL.md step 4.

## Site-wide: sticky header detached past ~1 viewport of scroll (root cause, not per-page)

- **`index.html`'s inline style had `html, body { height: 100% }` and `#root { height: 100% }` — this is the actual reason `NavBar`'s `sticky top-0` header would work for a short scroll distance and then detach and scroll away, on ANY page long enough to exceed one viewport height.** `position: sticky`'s "how far can it stay stuck" range is bounded by its own DOM parent's content box, not by the visual/scrollable page — with `#root` (the sticky header's direct parent, since React fragments don't add a DOM node) pinned to exactly `100%` (= 100dvh) while its actual content (especially a 5000+px-tall case-study canvas) overflows past that via `overflow: visible`, the header only had roughly `(one viewport − header height)` px of stuck range before falling off the bottom of its own undersized parent. This had been latent since before this session (the page's own `<meta description>` even claims "a sticky navbar" as a feature) — it only became obvious once a page long enough to scroll well past one viewport (any case study) was scrolled far enough to hit the ~818px-ish breaking point, which nobody had tested that deep before. **Fix: `height: 100%` → `min-height: 100%` on `html`, `body`, and `#root`** — keeps the "fill at least the viewport" guarantee (useful for short pages) while letting the box grow to real content height so sticky has the full page to stay stuck in. Check `index.html`'s inline `<style>` tag first (not `tailwind.css`) if a sticky/fixed element mysteriously detaches after a fixed amount of scroll on any future page — this is a global HTML-level constraint, invisible from component code.
- **A second, unrelated-looking bug found at the same time, also NavBar-specific: the sticky header's own colored pill bar (`bg-[#d7b8ff]`) is deliberately narrower than the full viewport (`md:w-[632px]`/`lg:w-[932px]`, not full-width) — the header's own outer padding/gutter area around that pill has no background at all.** On the Home page this never mattered (nothing but normal page flow sits directly under the header at scroll≈0). On case-study pages, the `FrameWrapper`-scaled raw Figma canvas has absolutely-positioned text that can land at any y-coordinate, including right under the header — since the header's surrounding area is transparent, that stray text shows through, looking like content "poking out" above/beside the pill. **Fix: give the `<header>` itself (not just the inner pill) an opaque page-background color (`bg-[#fff3ff]`)** so the whole top strip masks anything scrolling underneath, regardless of the pill's own narrower width. Because every page already uses this same `#fff3ff` background, this is invisible/seamless on Home and only functionally matters on case-study pages — don't mistake the fix for a visual change if you're comparing before/after screenshots on Home.

## Breakpoints (defined in `src/styles/tailwind.css` under `@theme`)

- `md` triggers at **744px**, styled for an 834px tablet canvas (8 col / 44 margin / 22 gutter).
- `lg` triggers at **1200px**, styled for a 1440px desktop canvas (12 col / 60 margin / 24 gutter). When Patricia says "desktop" or references 1440px, she means `lg:` classes in this codebase — the trigger width (1200) and the design canvas it represents (1440) are different numbers, don't conflate them.
- No prefix = mobile base. Her base-canvas numbers (e.g. a 400px-wide card) assume a canvas around ~440px — noticeably wider than the common 375/390px phone viewports used for testing.
- These are custom, not Tailwind defaults — don't reach for the usual 640/768/1024/1280 breakpoint assumptions in this repo.
- **Fixed Figma-canvas pixel sizes on a container (card width/height) must be applied as `w-full max-w-[Npx]`, never a flat `w-[Npx]`.** Every breakpoint's trigger width is meaningfully narrower than the canvas it's styled for (mobile ~440 vs common 375 devices is the widest gap; tablet 834 vs common 768 iPads; desktop 1440 vs the 1200 trigger). A flat fixed width overflows the viewport on real, common device widths — and since the site root has `overflow-x-clip`, that overflow doesn't even show as a scrollbar, it just silently clips content off-screen with no visual warning. `max-w` lets the box shrink to fit narrower real viewports while still hitting the exact spec size wherever the viewport allows it.
- **Corollary bug, found in the About-card fix: a base `max-w-[Npx]` with no `lg:` prefix silently caps a later `lg:w-[Mpx]` when M > N.** `max-width` always wins over `width` at the same specificity regardless of source order, so `w-full max-w-[440px] ... lg:w-[647px]` renders at 440px wide at desktop, not 647 — the `lg:w-` looks like it should override but `max-w-[440px]` (unprefixed, so still active at `lg`) clips it back down. Whenever a component's `lg:` width is bigger than its base `max-w`, add a matching `lg:max-w-none` (or `lg:max-w-[Mpx]`) in the same edit, and verify with `getComputedStyle(el).width` — the rendered size can look "close enough" at a glance and hide the cap.

## Accessibility (WCAG AA)

- **`#9b72ce` (the "accent purple" used for headings/labels) only passes AA contrast (4.5:1) against this card's light lavender background when the text is large and bold enough to qualify for the "large text" 3:1 exception** (≥18.66px at `font-extrabold`/700+, or ≥24px at any weight) — measured contrast is only ~3.4:1, which fails the normal-text 4.5:1 threshold. This is why the case-study card title is `font-extrabold` at every breakpoint and was bumped to `1.25rem` (20px) at mobile specifically — smaller/lighter uses of this color need a different fix (see next point). `#543976` (the darker "body" purple), by contrast, passes AA (~8.7:1) at any size — prefer it for small/regular-weight text.
- The meta line's "Role:"/"Team:" label span used to be `#9b72ce`, which fails AA at 12-18px regular weight (not bold, and below the large-text size threshold) — this wasn't mobile-specific, it was failing at every breakpoint, just not noticed until mobile accessibility was actually checked. Changed to `#543976` at all breakpoints (same markup renders everywhere), which makes the label and value the same color now — previously they were two shades of purple for visual distinction. If Patricia wants that distinction back, do it via weight (e.g. `font-semibold` on the label) rather than reverting to the non-compliant color.
- **Superseded:** `TagPill`'s old white-on-purple combo (`text-[#fff3ff]` on `bg-[#b488eb]`) measured ~2.6:1 — failed AA even at large sizes. This was fixed twice: first as a stopgap (darkened text to `#3e2859`, kept `#b488eb`), then properly once Patricia gave the real Figma node (`Pill`, node-id `457:112` in the Workshop file) — the actual spec is `bg-[#825db1]` (darker violet) + `text-[#fff3ff]` (white), which measures ~4.7:1 and passes AA on its own with no compromise needed. See "Design tokens" below for the current spec; don't reintroduce `#b488eb`.
- `GradientButton`'s text color (`#543976`) against its `from-[#fad89e] to-[#f29bfd]` gradient passes AA at both ends of the gradient (~7.1:1 peach end, ~5.0:1 pink end) — checked, no fix needed.
- When bumping a small/clipped mobile description for legibility, prefer `line-clamp-N` over letting it wrap freely — WCAG doesn't set a minimum font size, but readability guidance (Apple HIG / Material) suggests not going below ~11-12px; capping line count keeps the height predictable so a legibility bump doesn't silently blow the fixed card height (see below).
- **When you bump a font size for accessibility, re-check the *tallest-content* card the same way you would for any spec change** — a mobile-only title going 15px→20px doesn't touch the row's height (it's still dominated by the fixed-size mockup), but if the row's own text can wrap to more lines at a given width, clamp it too (title got `line-clamp-2` here) or the extra line height shows up as real overflow at the narrower end of the responsive range, not at the design's true target canvas width.

## Units

- Patricia's px→rem conversions assume a 16px root font size (the browser/Tailwind default, unchanged in this project): `Npx / 16`. Common ones used so far: 24px = `1.5rem`, 18px = `1.125rem`, 14px = `0.875rem`, 12px = `0.75rem`.
- She's asked for rem on **tablet and mobile** text-size tweaks so far, leaving desktop as plain `px`. Don't convert desktop's sizes to rem unless asked — match whatever unit she gives for that specific breakpoint, per element. Mobile also needed one non-literal value: 13px = `0.8125rem` (tags/button), since 13 isn't a clean sixteenth — rem conversion doesn't require a "nice" fraction, just `Npx / 16`.
- `TagPill` (shared component — also used in `AboutSection`'s skill pills) didn't have a way to override its font size per call site, so it now takes an optional `textSize` prop (default = its original `text-[13px] md:text-[14px]`, so every other call site is unaffected). When a shared component needs a one-off breakpoint-specific tweak for just the case-study card, add a prop rather than editing the component's hardcoded classes directly — otherwise the change silently leaks into every other place that component is used (About section pills, in this case).
- Reminder on why the card needs an explicit `lg:` value whenever `md:` changes for text sizes: with no `lg:` override, `lg:` inherits whatever `md:` says (that's how the cascade already bit us with the mockup sizes/card dimensions earlier). Every `md:text-[...]` tweak in the card needs a same-line `lg:text-[...]` restating the desktop value, or desktop's text silently shrinks/grows along with tablet's.

## Recurring root cause

- The site was rebuilt from Figma exports into responsive Tailwind components **by hand**, not auto-generated. Decorative elements that don't map cleanly onto a fluid layout (absolute-positioned blur blobs, fixed-size cards, animated background shapes) tend to get **dropped during the responsive rebuild** because they only really made sense at one fixed canvas size.
- When Patricia reports something "missing" on one specific breakpoint (usually desktop), check the closest raw Figma export under `src/imports/` first — the missing piece is often already coded there, just never ported into the live responsive component. Port and adapt it rather than redesigning from scratch.
- This repo has **no 1440-canvas Figma export checked in** — only 1024px raw exports exist (`src/imports/Home`). When exact pixel coordinates/sizes for a desktop-only decorative element aren't available at the real canvas size, proportionally adapt from the closest available export and verify visually against whatever screenshot Patricia provides, rather than guessing blind.
- Animated decorative elements exported from Figma (e.g. rotate/resize/translate keyframes) are usually given as a single 0%→100% transition (Figma smart-animate output), not a loop. For an ambient background effect, convert to `animation: ... infinite alternate` so it ping-pongs continuously — a one-shot animation looks broken/frozen after it finishes.
- **A "gradient border" simulated via a nested-div padding trick (outer div = gradient background + padding, inner element = solid fill covering the center) breaks silently if the outer padding isn't equal on all four sides — the visible "border" thickness is literally whatever padding is left uncovered per side.** `TagPill` briefly had `px-[2px] py-[8px]` on the outer wrapper (probably pasted/adapted from a differently-proportioned element), producing a border that looked 4× thicker top/bottom than left/right — this is easy to miss by eye at a glance but obvious once you inspect `getComputedStyle(outer).padding`. When a "gradient border" pill/card looks lopsided, check the wrapper's padding shorthand for a `px`/`py` split before assuming the border-radius or corner math is wrong — the fix is making the padding a single uniform value (equal to the stroke weight), not adding more div nesting.
- **`get_design_context`'s codegen silently flattens a gradient *stroke* down to a solid `border` using just the gradient's first color stop — it does not warn that it dropped the gradient.** For `TagPill`/`Pill` (node `457:112`), the generated code returned `border-3 border-[#fad89e] border-solid`, which looks like a complete, trustworthy answer — but the real Figma Inspector panel showed the stroke is `Gradient/400` (linear, `#fad89e` → `#f29bfd`), weight **2** (not 3), position "Inside". Trusting the generated CSS literally produced a visually-wrong solid-peach border at the wrong thickness. **Whenever a stroke/fill in the generated code is a flat hex and the element looks even slightly like it has a soft/multi-tone edge in the reference screenshot, ask Patricia to screenshot the Inspector's Fill/Stroke panel (not just the canvas) before trusting the codegen's color** — this is a known lossy conversion, not a one-off fluke, and it also silently drops the exact stroke *weight* (defaults to whatever `border-N` Tailwind scale step is closest) along with the gradient. The recovered spec here: outer wrapper padding = stroke weight (`p-[2px]`, not `p-[3px]`), inner radius = outer radius − weight (`38px` = `40px` − `2px`), gradient direction/stops read straight from the Inspector's gradient editor (`FAD89E` at 0% → `F29BFD` at 100%, linear/left-to-right — matches the site's existing `from-[#fad89e] to-[#f29bfd]` token, so no new gradient direction to figure out).

## Component map (About me section)

- `AboutSection` (`src/imports/Home-1/index.tsx`) — description card, stats, and photo+pills all live here now; `StatsRow` (separate section, right after `AboutSection`) still renders the horizontal stats layout for base/tablet but is `lg:hidden`, since desktop shows a vertical stats block *inside* `AboutSection` instead (duplicated render, same pattern as other base-vs-desktop content swaps in this file — cheaper than fighting document flow to reposition one shared section).
- **Figma gives independently-measured margins per element (not a shared row gap) — do the arithmetic before assuming two elements can't overlap.** Patricia's spec had the card at `margin-left:172/width:647` and the photo frame at `margin-left:569/width:699/margin-right:172` on the *same* 1440 canvas — 172+647=819 and 569+699=1268, so the two boxes overlap by 250px (819−569). This wasn't a mistake; she confirmed the photo frame is intentionally the background layer, overlapping the card by exactly that much. Rule of thumb: when two elements' independent margin-left/width numbers overlap, check whether one is meant to sit behind the other (z-index) before assuming the spec is inconsistent or belongs to a different (wider) canvas.
- **Translating independent Figma margins into flex requires one shared inset (`pl-[Npx]`) plus a negative margin on the second item, not a flex `gap`.** Card at x=172 and photo at x=569 on a 1440 canvas, with the page's own margin already at 60 (`Container`'s `lg:px-[60px]`): the row needs an extra `lg:pl-[112px]` (172−60) so the first item starts at x=172, then the second item (which would naturally start at 172+647=819 with zero gap) needs `lg:-ml-[250px]` to pull it back to its real x=569 (819−569=250). This reproduces Figma's independent-absolute-position layout inside a normal flex row without abandoning flex for true `position:absolute`.
- **A flex item with a later DOM position but earlier `order` (or vice versa) needs an explicit `z-index` to paint on top — flex paints in `order`, not DOM, order.** Here the photo frame (`lg:order-2`) would otherwise paint *above* the card (DOM-first, no order) once they overlap, hiding the card text. Fix: `lg:z-10` + `lg:relative` on the card. Same root cause already noted for `DualPhoneMockup`'s two devices.
- **A section-wide heading must NOT be a flex item inside the same row it labels, if that row also has a custom `pl-[Npx]` inset for its own children.** First attempt put the "About me" `<h2>` as a sibling flex item inside the same `lg:flex-row` as the card+photo columns — the row's padding meant for those two columns pushed the heading right too, and it word-wrapped fighting for space against the card. Fix: put the extra inset (`pl-[112px]`, `gap-0`) on an *inner* wrapper div (`contents` at base, `lg:flex` at desktop) that holds just the two columns, leaving the heading as a plain block-level sibling above it, unaffected by the inset and free to stay centered/full-width at every breakpoint.
- **Tailwind v4's separate-transform utilities (`rotate-*`, `translate-x/y-*`) compile to standalone CSS `rotate`/`translate` properties, not a combined `transform` value** — inspecting `getComputedStyle(el).transform` here read back `"none"` even though the rotation/translation were visibly applied; check `getComputedStyle(el).rotate` / `.translate` instead when debugging why a rotated/translated element "looks right but computed transform says none".
- **When a box is rotated 90/-90deg and then measured, don't pre-swap its width/height classes to match the *post-rotation* footprint — Tailwind (and CSS) rotation swaps the bounding box for you.** The About-card's blur-motion wrapper needed a final on-screen footprint of ~618×206 (wide, short) from a Figma-given *pre-rotation* box of 206.495×618.079 rotated -90deg. Writing the classes as `w-[206.495px] h-[618.079px] -rotate-90` is correct; writing `h-[206px] w-[618px] -rotate-90` (i.e. already-swapped dimensions plus the same rotation) cancels out and reproduces the *original* (wrong) footprint — easy mistake since both numbers "look like they're in the class list", just swapped from what the non-rotated intuition expects.
- The About-card blur-motion (`.blur-motion-about-a/b` in tailwind.css) reuses the *same* `--blur-motion-rotate-a/b` custom properties and `blur-motion-a/b-rotate` keyframes as the case-study cards' `BlurMotionBackdrop` — only the shape keyframes (height/width/translate deltas) differ per instance. When Figma gives a new blur-motion animation spec, check whether its rotate range matches an existing `--blur-motion-rotate-*` property before adding a new one; the rotate ranges (90°→180° and -180°→-360°) are shared, only the size/translate numbers are per-element.
- **The stats (`+4`/`3`/`+6`) render twice in the DOM** — once inside `AboutSection` (`hidden lg:flex`, desktop-only vertical column) and once in the separate `StatsRow` section (`lg:hidden`, base/tablet horizontal row). A plain `document.querySelectorAll` text match during verification will find both; check `getComputedStyle(el.parentElement...).display` (or just resize to the breakpoint you mean) before trusting which one you're reading — don't assume the first DOM match is the visible one. When Patricia gives a style fix for these labels without naming a breakpoint, default to whichever instance she's currently looking at in this conversation (usually the one most recently touched) rather than applying it to both — the two instances have historically had different sizes (13/14px tablet-scaled vs 18px desktop) and there's no evidence they're meant to match.
- `preview_screenshot` can return a stale/frozen frame right after a `window.scrollTo`/`scrollBy` call on this site — `<html>` has `scroll-behavior: smooth` globally, so an unmodified `scrollTo` animates asynchronously and the screenshot can fire mid-animation (or before it even starts). Set `document.documentElement.style.scrollBehavior = 'auto'` (or pass `behavior: 'instant'`) before scrolling in a verification pass, and prefer computing an absolute target (`window.scrollY + el.getBoundingClientRect().top - offset`) over repeated relative `scrollBy` calls, which compound oddly if a prior smooth-scroll animation hasn't settled yet.
- **"El motion está en el frente, quiero que esté detrás" (About-card blur-motion) turned out to be about overflow bleed, not z-index vs. a sibling.** The blur's individual ellipses animate their own size via keyframes (`blur-motion-about-a/b-shape`) past the wrapper's nominal 618×206 footprint at some points in the loop; nested inside the card with no `overflow-hidden`, that growth visibly spilled past the card's rounded edge into the stats block below, reading as "in front of"/floating beside the card rather than a glow confined behind its glass. First instinct was to fix this via stacking order (move the blur to be a z-index sibling of the card instead of a trapped descendant) — that's real CSS behavior (a negative-z-index descendant of a `z-10` ancestor can only paint behind that ancestor's *own* content, never behind the ancestor's siblings) but turned out to be the wrong diagnosis here, since in this card's actual geometry the blur's static bounding box (618×206) fits entirely inside the card's box (647×223) with no static overlap into neighboring elements — so it was never actually escaping sideways into a sibling, only bleeding vertically past the card's own edge during animation. The real fix: keep the blur nested inside the card (simpler, and its 50%/50% self-centering needs the card as the direct positioned ancestor anyway) and add `overflow-hidden` to the card itself, clipping any animated overshoot to the card's rounded silhouette. Lesson: when a decorative element "looks like it's in front of X," check whether X is a geometric sibling it could actually be escaping into, vs. whether the element is simply bleeding past its own container's edge — `overflow-hidden` on the immediate container is the cheaper, more targeted fix and was sufficient here.

## Tablet (`md:`) follow-up to the About-card fixes

- **The About section's tablet (`md:`) treatment mirrors desktop's almost exactly** (scattered floating pills instead of the flowing wrap list, same 699×778 photo frame, same 456×714 image at the same 178/64/65/0 margins, same card blur-motion) — the only real differences are: no side-by-side overlap (tablet stacks card above photo in normal flow, no `-ml-[250px]` trick) and the numbers are the *un-scaled* desktop numbers reused as-is at the smaller tablet canvas (699-wide photo frame on an 834 canvas is intentionally almost edge-to-edge, not proportionally shrunk). When Patricia says "el mismo blur motion del breakpoint de desktop" for a new breakpoint, default to literally reusing the same component/values (just add the new breakpoint's prefix alongside or instead of the old one) rather than re-deriving a scaled-down version.
- **A breakpoint-scoped extra inset/margin (e.g. `md:ml-[4px]`) silently cascades up to every wider breakpoint unless explicitly cancelled.** Tailwind's `md:` utilities apply at `md` *and above* (min-width cascade) — adding `md:ml-[4px]` to fix a tablet-only 4px discrepancy also shifted the desktop layout by the same 4px, since desktop had no `lg:ml-*` of its own to override it back. Same issue hit the photo frame's `md:mx-auto` leaking into desktop and fighting with its existing `lg:-ml-[250px]` (mixed auto/fixed margins on opposite sides of one box behave unpredictably). Rule: whenever a fix only targets one specific breakpoint by giving it a property no adjacent breakpoint had before, check whether a *wider* breakpoint already sets something for that same property — if not, it inherits the new value, so add an explicit `lg:`-reset (`lg:ml-0`, `lg:mx-0`, etc.) in the same edit and re-verify the wider breakpoint's computed position, not just the one you're actively fixing.
- **Figma's independent absolute margin-left value can be *smaller* than what it looks like it should reconcile to** — the tablet card's Figma margin-left (48px) is 4px more than the page's own `md:px-[44px]` margin, not a clean multiple of the grid. Rather than assume a typo/rounding and default to 44, trust the literal number and express it as the page margin *plus a small delta* (`md:ml-[4px]` here, extra beyond Container's own padding) — same "additional local inset beyond the page margin" pattern as desktop's 172 (=60+112), just a much smaller delta this time.
- **A hard `max-w-[Npx]` cap for a `justify-between` row must go on a child *inside* `Container`, not appended to `Container`'s own className.** `Container` already bakes in `max-w-[1440px]` + responsive `px-*` page-margin padding on its own root div; adding another `max-w-[746px]` directly to that same div would apply *on top of* its own padding, double-subtracting the margin (746 cap, then −88 more from the still-active `md:px-[44px]` inside it = 658 rendered, not 746). Wrap the row in its own inner `div` with the `max-w-[Npx] mx-auto` instead, so the cap applies to content *inside* Container's padding, matching what Patricia's Figma number actually measures (content-area width, not outer-box width).
- **`justify-between` + `shrink-0` + a `max-w` cap fixes "items wrap to a vertical stack on real (narrower-than-canvas) tablet viewports"** — the previous stats row used a fixed `gap-x-[48px]` with `flex-wrap`, which only fit on one line at exactly the 834 design canvas; any real tablet narrower than that (e.g. 768px iPads) forced wrapping into a vertical stack. Swapping to `flex-nowrap md:justify-between` (space distributed by the container, not a fixed gap) plus `shrink-0` on each fixed-width item keeps it one row and non-overflowing across the whole `md` range without needing per-viewport tuning.

## Mobile follow-up to the About-card fixes

- **The About section now uses the same scattered-pill/blur-motion treatment at all three breakpoints** — mobile (400×390 photo frame), tablet (699×778), desktop (699×778, overlapping the card). The old "flowing wrap tag list" (`SKILL_PILLS` + its `<TagPill>` map) was mobile-only dead weight once mobile also switched to scattered pills; deleted both the JSX block and the now-unused `SKILL_PILLS` constant rather than leaving it as unreachable code.
- **Scaling a fixed-coordinate absolute-position group (`SCATTERED_PILLS`, calibrated for the 699×778 tablet/desktop frame) down for a smaller breakpoint must scale around the *original* coordinate system's own box, not around the smaller frame's box.** First attempt used `absolute inset-0 scale-[0.5]` on the pills wrapper — `inset-0` stretches the wrapper to match its parent (400×390 at mobile), so the default center-origin scale pivoted around (200,195) instead of the coordinates' true design-space center (~350,390 in a 699×778 box). Pills with large raw offsets (e.g. `left:609`, calibrated to sit near the 699-wide frame's right edge) ended up scaled from the *wrong* pivot and drifted past the actual (much narrower) mobile viewport's right edge. Fix: give the wrapper an explicit, unscaled `w-[699px] h-[778px] left-0 top-0` (the *original* coordinate space, ignoring the actual smaller frame it's visually inside) plus `origin-top-left`, so scaling shrinks the whole 699×778 layout uniformly from its own top-left corner — which also happens to be where the frame's own top-left corner is, at every breakpoint. Same root technique as `BlurMotionBackdrop`'s responsive `scale-[0.41] md:scale-[0.75] lg:scale-100` (documented above), but that case didn't hit this bug because its wrapper was already given explicit non-stretched dimensions from the start.
- **`AboutBlurMotion` and the card's `overflow-hidden` containment apply at every breakpoint now** (previously desktop-only, then tablet+desktop) — when Patricia says "el mismo blur motion" for a new breakpoint, the answer is almost always "remove the breakpoint qualifier so it's unconditional," not "add another prefixed variant."

## Case study detail pages — broken font-family override (AfterStory)

- **`font-['Momo_Trust_Sans:WEIGHT',sans-serif]` (WEIGHT = Regular/Medium/Bold/ExtraBold) is a non-functional font-family override, not a real per-weight font selector — found on 57 elements across `CaseStudyAs/index.tsx`, silently rendering as fallback `sans-serif` on every one.** Root cause: `tailwind.css` registers exactly one `@font-face` — `font-family: "Momo Trust Sans"` (a single variable font, `font-weight: 200 800`) — and `--font-sans: "Momo Trust Sans", ui-sans-serif, system-ui, sans-serif` is the global default every element already inherits. The Figma-export naming convention bakes the weight into the family string itself (`"Momo Trust Sans:Medium"` etc.), which matches no registered `@font-face`, so the browser falls through to the last item in that family's own list — generic `sans-serif` — instead of falling back further to the correct inherited global default. **Fix: delete the entire broken `font-['Momo_Trust_Sans:WEIGHT',sans-serif]` fragment (or its combined form `font-['Momo_Trust_Sans:WEIGHT','Noto_Sans_Symbols2:WEIGHT',sans-serif]`) wherever it appears — never just edit the weight name, since none of them work.** Safe to strip file-wide in one pass (verified via `sed -E`, then `grep -c "Momo_Trust_Sans:"` → 0): the numeric weight is always *also* set by a separate, correctly-working Tailwind class (`font-extrabold`/`font-bold`/`font-medium`/`font-normal`) right next to the broken one, so removing the broken fragment loses nothing — the correct family cascades in from the global default and the weight is preserved. Check any *other* case study file (`CaseStudyCove`/`CaseStudyLuh`) for the same pattern before assuming this was AfterStory-only — it's a shared Figma-export-codegen habit, not specific to one file.

## Case study detail pages — mobile breakpoint (AfterStory, Level Up Habits)

- **Reading alignment intent straight from the raw Figma export's existing classes is a reliable shortcut when there's no mobile mockup for that specific case study.** Each section heading's container already carries a literal `text-right`/`text-center` class (or neither = left) in the desktop source that matches the section's intended alignment almost exactly — used this to confirm AfterStory's sections against its PDF reference, and to fill in Level Up Habits' alignment (no PDF given for it) by cross-checking the same pattern in `src/imports/CaseStudyLuh/index.tsx`. Cheap to grep for before asking the user or guessing.
- **The same-named "flavor" of a component (e.g. `MachineLearningStreamlineDjakarta`, `DesignerWorking1StreamlineDjakarta`) can be `absolute`-rooted in one case-study export and `relative`-rooted in another — never assume based on what a same-named component did in a sibling file.** AfterStory's `MachineLearningStreamlineDjakarta` is `relative` (safe, direct-scale reuse) while Level Up Habits' is `absolute h-[...] left-[...] top-[...]` (needs the crop-offset treatment); the reverse was true for `DesignerWorking1StreamlineDjakarta`. Check each file's own copy's root className before picking `MobileIllustrationCrop` (absolute-rooted, needs x/y/w/h crop math) vs. a plain resize wrapper (relative-rooted, just needs `transform: scale()`).
- **A `flex-[1_0_0]` chain silently collapses to 0×0 outside the exact nested-flex ancestor depth it was authored for.** AfterStory/LUH's `Bio2`/`Frame30`/`Frame35`/`UserPersona2` (the "Buyer persona" card) is `flex-col flex-[1_0_0] ... self-stretch` — `flex-[1_0_0]` sets `flex-basis:0`, which needs a flex ancestor with a *definite* height to resolve against; the original desktop composition happens to supply one several levels up, but reusing the component standalone (even wrapped in a plain `<div className="flex w-full">`) breaks that chain and the whole card renders at zero size (confirmed via `getBoundingClientRect()` → `{w:0,h:0}`, not just visually thin). `UserPersona1`("User persona")/`Bio1` happened to render fine standalone because their own wrapper (`Frame29`) is a plain row (`shrink-0 w-full`, not `flex-[1_0_0]`) — but don't take that as proof the pattern is generally safe; the Buyer-persona-specific card was still broken. **Fix used for both case studies: don't reuse `Bio`/`Bio1`/`Bio2`/`UserPersonaN` for mobile personas at all — build a single from-scratch `MobilePersonaRow` (photo circle + name/age/description card, `photoSide: "left"|"right"` prop, plain flex row/`flex-1`+`min-w-0` text box, no `flex-[1_0_0]` anywhere) and pass in the raw text/photo per persona.** Cheaper and more robust than debugging which specific reused sub-component will or won't collapse.
- **`scaleX(-1)` combined with `transform-origin: 0 0` (or any non-center origin) flips the element into negative coordinate space, not in place — it silently renders off-screen/clipped by the nearest `overflow-hidden` ancestor.** The Cove-style crop/scale helpers use `transform-origin: 0 0` deliberately (so a *separate* crop-offset translate lines up against a known corner) — copying that same origin into a *flip* transform breaks, since mirroring around the origin sends the whole box to the mirror side of x=0 instead of mirroring in place. Fix (see `MobileScaledIcon` in both `CaseStudyAs/index.tsx` and `CaseStudyLuh/index.tsx`): do the resize scale in one wrapper with `transform-origin: top left` (so the scaled box's top-left corner lands at the crop window's origin, matching its parent's size exactly with no overflow), then apply `scaleX(-1)` with `transform-origin: center` in a *separate* outer wrapper sized to exactly match that already-correctly-scaled box — flipping a box around its own center, once the box's edges already coincide with the visible window, mirrors it in place instead of off dose-screen. Symptom when this goes wrong: `getBoundingClientRect()` on the flipped element reports the right *width/height* (scale math was fine) but a `left`/`top` far outside the visible viewport (often negative) — that mismatch (right size, wrong position) is the signature of an origin/flip bug, not a sizing bug.
- **A stale previously-loaded page can keep responding to `preview_eval`/`preview_screenshot` calls even after the dev server's code changes and a `window.location.hash` update** — if an earlier debugging session navigated the preview tab directly to a static asset (e.g. a temp reference image copied into `public/` for viewing, since deleted), later `window.location.hash = "#..."; location.reload()` calls reload *that same static asset* (hash intact, path unchanged), not the app's `index.html`. Symptom: `window.innerWidth` reads the image's native pixel width, `document.querySelectorAll('svg').length` is 0, `document.title` is the filename — none of which resemble the app. Fix: `window.location.href = "http://localhost:<port>/#<route>"` (a full path navigation, not just a hash mutation) to force back onto the actual app entry point. Worth an early sanity check (`location.href`/`document.title`) whenever a screenshot looks unexpectedly wrong after resuming a long session.
- **A user's literal verbal alignment instruction can be deliberately overridden by her own follow-up reasoning even when the raw source code disagrees with both.** Patricia's original message said Level Up Habits' "Future Feature" section should go right-aligned (explicitly reasoning: LUH has no User Testing section, so the alternating pattern collapses and both Future Features *and* Key Learnings go right) — but LUH's actual `Frame54` (Future Features) source has no `text-right`/`items-end` class at all (default left), unlike `Frame55` (Key learnings) which does. Went with her stated reasoning over the source's literal left-default for this one section, since she gave an explicit, deliberate explanation rather than it being ambiguous — same category of override as the AfterStory Archetype photo-side decision (resolved via `AskUserQuestion` earlier in that task), just resolved directly from her own words this time since no visual reference contradicted it.
