# Fix-design learnings — Responsive UX/UI Portfolio

Living reference for the `fix-design` skill. Update in place, don't append a changelog — see SKILL.md step 4.

## Live vs. legacy files

- The live homepage is `src/imports/Home-1/index.tsx`, wired up via `src/app/App.tsx` (`import Home from "@/imports/Home-1"`).
- `src/imports/Home/index.tsx` (no `-1`) is a **legacy raw Figma export** at a fixed 1024px canvas — not imported anywhere live. Treat any `src/imports/<Name>` without a page wrapper importing it in `src/app/` as reference-only.
- Case study pages (`src/app/CaseStudy.tsx`, `CaseStudyLuh.tsx`, `CaseStudyCove.tsx`) each render a raw Figma export (`src/imports/CaseStudyAs`, `CaseStudyLuh`, `CaseStudyCove`) inside a `FrameWrapper` that scales the whole pixel-exact frame — these ARE live, unlike `imports/Home`. Don't assume every `src/imports/*` export is dead code; always trace from `App.tsx` first.

## Breakpoints (defined in `src/styles/tailwind.css` under `@theme`)

- `md` triggers at **744px**, styled for an 834px tablet canvas (8 col / 44 margin / 22 gutter).
- `lg` triggers at **1200px**, styled for a 1440px desktop canvas (12 col / 60 margin / 24 gutter). When Patricia says "desktop" or references 1440px, she means `lg:` classes in this codebase — the trigger width (1200) and the design canvas it represents (1440) are different numbers, don't conflate them.
- No prefix = mobile base. Her base-canvas numbers (e.g. a 400px-wide card) assume a canvas around ~440px — noticeably wider than the common 375/390px phone viewports used for testing.
- These are custom, not Tailwind defaults — don't reach for the usual 640/768/1024/1280 breakpoint assumptions in this repo.
- **Fixed Figma-canvas pixel sizes on a container (card width/height) must be applied as `w-full max-w-[Npx]`, never a flat `w-[Npx]`.** Every breakpoint's trigger width is meaningfully narrower than the canvas it's styled for (mobile ~440 vs common 375 devices is the widest gap; tablet 834 vs common 768 iPads; desktop 1440 vs the 1200 trigger). A flat fixed width overflows the viewport on real, common device widths — and since the site root has `overflow-x-clip`, that overflow doesn't even show as a scrollbar, it just silently clips content off-screen with no visual warning. `max-w` lets the box shrink to fit narrower real viewports while still hitting the exact spec size wherever the viewport allows it.

## Accessibility (WCAG AA)

- **`#9b72ce` (the "accent purple" used for headings/labels) only passes AA contrast (4.5:1) against this card's light lavender background when the text is large and bold enough to qualify for the "large text" 3:1 exception** (≥18.66px at `font-extrabold`/700+, or ≥24px at any weight) — measured contrast is only ~3.4:1, which fails the normal-text 4.5:1 threshold. This is why the case-study card title is `font-extrabold` at every breakpoint and was bumped to `1.25rem` (20px) at mobile specifically — smaller/lighter uses of this color need a different fix (see next point). `#543976` (the darker "body" purple), by contrast, passes AA (~8.7:1) at any size — prefer it for small/regular-weight text.
- The meta line's "Role:"/"Team:" label span used to be `#9b72ce`, which fails AA at 12-18px regular weight (not bold, and below the large-text size threshold) — this wasn't mobile-specific, it was failing at every breakpoint, just not noticed until mobile accessibility was actually checked. Changed to `#543976` at all breakpoints (same markup renders everywhere), which makes the label and value the same color now — previously they were two shades of purple for visual distinction. If Patricia wants that distinction back, do it via weight (e.g. `font-semibold` on the label) rather than reverting to the non-compliant color.
- **`TagPill`'s white-on-purple combo (`text-[#fff3ff]` on `bg-[#b488eb]`) measures ~2.6:1 — fails AA even at large sizes (needs 3:1 minimum).** This is a site-wide token issue (same component used in the About section's skill pills), not something fixable by resizing text on mobile. Flagged to Patricia rather than silently changing the shared brand color; needs either a darker pill background or darker/bolder text, her call.
- `GradientButton`'s text color (`#543976`) against its `from-[#fad89e] to-[#f29bfd]` gradient passes AA at both ends of the gradient (~7.1:1 peach end, ~5.0:1 pink end) — checked, no fix needed.
- When bumping a small/clipped mobile description for legibility, prefer `line-clamp-N` over letting it wrap freely — WCAG doesn't set a minimum font size, but readability guidance (Apple HIG / Material) suggests not going below ~11-12px; capping line count keeps the height predictable so a legibility bump doesn't silently blow the fixed card height (see below).
- **When you bump a font size for accessibility, re-check the *tallest-content* card the same way you would for any spec change** — a mobile-only title going 15px→20px doesn't touch the row's height (it's still dominated by the fixed-size mockup), but if the row's own text can wrap to more lines at a given width, clamp it too (title got `line-clamp-2` here) or the extra line height shows up as real overflow at the narrower end of the responsive range, not at the design's true target canvas width.

## Units

- Patricia's px→rem conversions assume a 16px root font size (the browser/Tailwind default, unchanged in this project): `Npx / 16`. Common ones used so far: 24px = `1.5rem`, 18px = `1.125rem`, 14px = `0.875rem`, 12px = `0.75rem`.
- She's asked for rem on **tablet and mobile** text-size tweaks so far, leaving desktop as plain `px`. Don't convert desktop's sizes to rem unless asked — match whatever unit she gives for that specific breakpoint, per element. Mobile also needed one non-literal value: 13px = `0.8125rem` (tags/button), since 13 isn't a clean sixteenth — rem conversion doesn't require a "nice" fraction, just `Npx / 16`.
- `TagPill` (shared component — also used in `AboutSection`'s skill pills) didn't have a way to override its font size per call site, so it now takes an optional `textSize` prop (default = its original `text-[13px] md:text-[14px]`, so every other call site is unaffected). When a shared component needs a one-off breakpoint-specific tweak for just the case-study card, add a prop rather than editing the component's hardcoded classes directly — otherwise the change silently leaks into every other place that component is used (About section pills, in this case).
- Reminder on why the card needs an explicit `lg:` value whenever `md:` changes for text sizes: with no `lg:` override, `lg:` inherits whatever `md:` says (that's how the cascade already bit us with the mockup sizes/card dimensions earlier). Every `md:text-[...]` tweak in the card needs a same-line `lg:text-[...]` restating the desktop value, or desktop's text silently shrinks/grows along with tablet's.

## Recurring root cause

- The site was rebuilt from Figma exports into responsive Tailwind components **by hand**, not auto-generated. Decorative elements that don't map cleanly onto a fluid layout (absolute-positioned blur blobs, fixed-size cards, animated background shapes) tend to get **dropped during the responsive rebuild** because they only really made sense at one fixed canvas size.
- When Patricia reports something "missing" on one specific breakpoint (usually desktop), check the closest raw Figma export under `src/imports/` first — the missing piece is often already coded there, just never ported into the live responsive component. Port and adapt it rather than redesigning from scratch.
- This repo has **no 1440-canvas Figma export checked in** — only 1024px raw exports exist (`src/imports/Home`). When exact pixel coordinates/sizes for a desktop-only decorative element aren't available at the real canvas size, proportionally adapt from the closest available export and verify visually against whatever screenshot Patricia provides, rather than guessing blind.
- Animated decorative elements exported from Figma (e.g. rotate/resize/translate keyframes) are usually given as a single 0%→100% transition (Figma smart-animate output), not a loop. For an ambient background effect, convert to `animation: ... infinite alternate` so it ping-pongs continuously — a one-shot animation looks broken/frozen after it finishes.

## Component map (case studies section)

- `CaseStudyCard` (`src/imports/Home-1/index.tsx`) is the shared card used for all three projects (Level Up Habits, AfterStory, Cove) — fix once here, not per-project.
- `BlurMotionBackdrop` (same file) holds the two ambient blur blobs behind the phone mockup, desktop-only (`hidden lg:block`). Keyframes live in `src/styles/tailwind.css` as `.blur-motion-a` / `.blur-motion-b`.
- `DualPhoneMockup` (same file) renders two devices per card: `bigImg` is the larger, angled/tilted background device (a single flattened photo-mockup PNG with its screen content already baked in — not swappable per-screen), and the small overlay (`smallFrameImgs` + `smallScreenImg`) is the smaller, straight-on **front-facing** phone, screen content composited separately. When Patricia says "el mockup vertical" or "el que mira al frente", she means this small overlay, not the big tilted one — its aspect (`aspect-[194/398]` ≈ 0.487) matches a real phone-screen recording's ratio, which is the tell.
- The front-facing overlay screen now supports video: `DualPhoneMockup` takes an optional `smallScreenVideo` and renders a muted/looping/autoplaying `<video>` (with the old `smallScreenImg` as `poster`) instead of the static `<img>` when provided. `CaseStudyCard` forwards this via its own optional `mobileScreenVideo` prop. All three cards now have their video wired in (`VIDEO_LUH_URL`, `VIDEO_AS_URL`, `VIDEO_COVE_URL` constants near the top of `Home-1/index.tsx`).
- `smallFrameImgs` is `[imgMobileMockupLuh, imgMobileMockupLuh1]` for every card — two near-identical phone-bezel PNGs (transparent screen cutout, rounded corners + Dynamic Island notch), not decorative duplicates. They must sandwich the screen content, not both sit behind it: index 0 renders behind the screen/video, index 1 renders **in front** of it (`pointer-events-none`, same `absolute inset-0`). The front one's rounded/transparent cutout is what masks the video's straight rectangular corners — with a static image this didn't matter much since the image itself had soft edges, but a `<video>` box is a hard rectangle and its corners will visibly poke out past the phone-screen shape unless the front bezel layer is painted after it in the DOM.
- **Desktop mockup sizing is fixed-px and content-sized, not proportional to the card.** Below `lg`, both devices still use the original fluid system (`bigAspect` on the outer box + `smallLeftPct/TopPct/WidthPct` percentages) so they scale with viewport — keep that for mobile/tablet. At `lg`, the outer box switches to `lg:flex lg:items-end` and both devices get literal Figma pixel sizes as flex items: front-facing vertical phone `lg:w-[194px] lg:h-[398px]` with `lg:ml-[70px]` (its offset from the card's mockup column), tilted side phone `lg:w-[308px] lg:h-[384px]` with `lg:-ml-[38px]` (pulls it left to overlap the vertical phone by 38px). The vertical phone keeps `z-10` so it paints in front of the tilted one regardless of DOM/flex order (flex paints in `order`-modified order by spec, so without an explicit z-index the later flex item would wrongly paint on top). The mockup column wrapper had to change from a proportional `lg:w-[45%]` to `lg:w-auto` (content-sized) once the devices got fixed pixel widths — a percentage column and fixed-px children fight each other (either wasted space or overflow into the text column), so once any mockup measurement is pinned to literal px at `lg`, check whether its parent column is still using a percentage width and switch it to `w-auto`/content-sized too.
- Getting this right took a few wrong turns worth remembering: don't assume a new width+height pair for `bigImg` is just "the same image, resized" — check the source PNG dimensions (`sips -g pixelWidth -g pixelHeight`) and look at the actual image (Read tool renders PNGs) before picking `object-contain` vs `object-cover`; a box whose aspect ratio doesn't match the source image's native aspect will letterbox (contain) or crop (cover), and guessing wrong here makes the mockup visually disappear or look cropped-wrong rather than erroring loudly. When Patricia's numbers don't reconcile with the current fluid/percentage layout, she's usually describing a fixed-px arrangement meant to fully replace the proportional one at that breakpoint, not a tweak to its inputs.
- Convention for binary assets too large/awkward to bundle via import (video, PDF): drop them in `public/<category>/` (e.g. `public/videos/`, matching the existing `public/cv/` pattern) and reference by absolute URL string constant, not a JS `import` — keeps big files out of the module graph, same as the CV download already does.
- Card height at desktop is a **fixed** `lg:h-[588px]` (not `min-h`) — Patricia confirmed she wants the card clamped to exactly 588px even though card content height varies per project (Cove has 2 meta lines vs. 1 for the others). The card already has `overflow-hidden`, so this clips rather than pushes the card taller; check the tallest-content card (currently Cove) after any change to the text column, since it's the one most likely to overflow first.
- Mobile card height went through **294px → 380px (too spacious) → 352px (nice at the ~440-460 canvas, but clipped at real 375px phones) → 416px (fits at both)**. The reason 352 wasn't enough: don't size a fixed mobile height by checking only the true target canvas width — the tag pills wrap the same way at 375px as at 460px, but with the card's `max-w` shrinking the available width, gaps/padding eat a proportionally bigger share, so the *narrow* device is the real worst case to measure against, not the canvas width. Once the height was sized for the narrow case, the canvas-width view had comfortable-but-uneven leftover space (all bunched below the button) — fixed by adding `h-full justify-center` to the mobile content wrapper (reset via `md:justify-normal` so it doesn't affect tablet/desktop) so any slack space distributes evenly above/below instead of pushing the button down. When "make it less cramped" is the ask and you don't have an exact target number, size to the narrowest realistic viewport's content need first, then use `justify-center` to absorb the extra room gracefully at wider ones — don't just pick a height that looks right at one width and hope.
- "View work" button: at desktop only, it detaches from the text-column flow and pins to the card's bottom-right corner — `lg:absolute lg:bottom-[36px] lg:right-[28px]` on the button, relying on the card's own `relative` for positioning. Below `lg`, it stays inline in the centered flow after the tags. When Patricia gives margin-right/margin-bottom numbers for an element she wants "aligned to the right/bottom" of a card, that's usually this pattern (Figma exports absolute-positioned nodes with margin-* offsets; translate to `right-[Npx] bottom-[Npx]` on an `absolute` element, not literal `margin-right`/`margin-bottom`, since the parent is a flex column here).
- **Base-only content grouping uses a `display: contents` toggle; tablet mirrors desktop, not mobile.** Patricia wants mobile specifically to show title+description *beside* the mockup, with Role/Team + tag pills *below* — different from desktop's side-by-side full-height columns (mockup | full text column). `CaseStudyCard` renders a small wrapper `<div className="flex flex-row ... md:contents">` around [mockup, an inline title/description block that's `md:hidden`] — from `md` up the wrapper disappears (`display: contents`) so its children rejoin the outer row (`md:flex-row md:items-center`) as if the wrapper wasn't there, and the *other* (hidden-until-`md`) title/description inside the main text column takes over instead, with that column set to `md:flex-1` so it shares the row with the mockup. First pass wrongly extended this compact mobile grouping up through tablet too (reasoning: the old smaller/fluid tablet mockup didn't fit a full stack) — but once tablet's mockup got its own literal Figma size (144×298, tall relative to the 408px card), it became clear tablet needed the *desktop* pattern (full-height side-by-side columns), not mobile's compact stacked block. When a breakpoint's mockup is tall enough to want the full card height, give it its own column rather than trying to squeeze title+description beside a small inline block.
- **Both mockups (front-facing vertical + tilted background) are literal Figma px at every breakpoint now** (mobile/tablet/desktop each with their own numbers) — the old `bigAspect`/`smallLeftPct`/`smallTopPct`/`smallWidthPct` percentage-based system is gone entirely (removed from `DualPhoneMockup`, `CaseStudyCard`, and all three `CaseStudyCard` call sites — don't reintroduce them). `DualPhoneMockup`'s root is `flex items-end`; the vertical phone is flex order-1 with `ml-[Npx]` (its absolute offset from the mockup column's left edge); the tilted one is order-2 with a *negative* margin computed as `(vertical's own absolute-offset margin-left value Patricia gave it) − (vertical's ml + vertical's width)` — i.e. when Patricia gives both mockups their own independent "margin-left" as if each were absolutely positioned from the same container edge (a common Figma-export idiom for overlapping layers), convert the second one's number into a flex delta-from-natural-position, which comes out negative (overlap) as long as the two boxes actually overlap in her spec.
- **A literal Figma height number can be tighter than the content it must hold — don't assume it'll just fit.** Fitting name+description+role/team+tags+button into `h-[294px]` (mobile) required capping the inline description to 2 lines (`line-clamp-2`, mobile only — tablet/desktop have their own full-width text column so don't need it), plus trimming gaps/padding that weren't part of Patricia's literal spec (card padding, inter-element gaps) since those were free to adjust. Always verify with the *tallest-content* card (Cove, 2 meta lines) — it fits with the least margin and is the first to clip if anything grows.
- `BlurMotionBackdrop` renders at every breakpoint (previously `hidden lg:block`, desktop-only) — Patricia confirmed the blur-motion blobs should behave the same everywhere. Its blob keyframes (`.blur-motion-a/b` in tailwind.css) animate between hardcoded pixel sizes tuned for the desktop mockup's scale, and a CSS animation's keyframe values override any static width/height class on the same element — so instead of trying to resize the blobs per breakpoint, the whole backdrop wrapper gets a responsive scale transform (roughly the ratio of that breakpoint's vertical-mockup height to desktop's 398px: currently `scale-[0.41] md:scale-[0.75] lg:scale-100`), shrinking everything inside proportionally. Recompute the ratio whenever the vertical mockup's height spec changes.
- The mockup footprint eats a bigger share of the card at narrower real viewports than at the exact spec canvas, same underlying issue as the card-width note above but for the mockups specifically — e.g. mobile's ~218px-wide mockup pair inside a 335px-wide card (at a real 375px phone) leaves very little room for the inline title/description. Verified it looks right at the true ~440px mobile canvas; narrower phones get a more cramped (but non-clipping, thanks to `line-clamp-2` and `max-w`) version. Flag this tradeoff rather than silently trying to fully solve it — it's inherent to literal fixed-px mockup specs at a canvas wider than common device viewports.
- Same narrow-viewport tradeoff shows up in the tag pills: at the true ~440px canvas, "UX Research"/"UI Design"/"Design Thinking" wrap to 2 rows already (3 pills rarely fit one line regardless of viewport) and the card height budget was tuned around that. At a real 375px phone the card itself shrinks (`max-w`), shrinking the tags row's available width further — it still wraps to the same 2 rows, so this specific case wasn't the actual regression risk; what *did* regress once when title/description sizes changed was the mobile row's own text height creeping past what fits before the second block even starts — always re-measure the *second* block (meta/tags/button), not just the row, after any mobile text-size change, since it's easy to assume the row is still the tall element when the mockup no longer dominates by as much a margin as before.
